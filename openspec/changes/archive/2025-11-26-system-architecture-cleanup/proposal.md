# 系統架構清理與性能優化提案

## Why
當前香港量化交易系統面臨三個關鍵問題，嚴重影響系統的可維護性、可靠性和性能：(1) 400+文件的過度複雜項目結構導致維護困難和開發效率低下；(2) Mock數據與真實數據混合使用造成數據質量問題和回測結果不可靠；(3) CPU利用率低、GPU加速配置複雜導致性能瓶頸。這些問題阻礙了系統的進一步發展和商業價值的實現，迫切需要系統性的解決方案。

## 概述

本提案旨在解決香港量化交易系統當前面臨的三個核心問題：過度複雜的項目結構、數據源管理混亂、以及性能瓶頸與資源浪費。通過系統性的架構清理和優化，將現有400+文件的複雜系統轉變為高效、可維護的簡化架構。

## 問題陳述

### 1. 項目結構複雜性問題
- **現狀**: 項目包含400+文件，大量重複腳本和測試代碼
- **影響**: 維護困難、開發效率低下、集成測試複雜
- **業務風險**: 新功能開發週期長，bug修復困難

### 2. 數據源管理混亂
- **現狀**: Mock數據與真實數據混合使用，多版本API共存
- **影響**: 回測結果不可靠，環境不一致，數據質量難保證
- **業務風險**: 量化策略基於不準確數據，交易決策可靠性低

### 3. 性能瓶頸問題
- **現狀**: CPU使用率低，GPU加速配置複雜，緩存機制不完善
- **影響**: 策略優化耗時長，硬體資源浪費，實時響應慢
- **業務風險**: 錯失市場機會，競爭力下降

## 解決方案

### 1. 項目結構簡化
以Simplified System為核心，建立清晰的模塊化架構：

```
CODEX--/
├── simplified_system/          # 核心簡化系統 (主要開發區域)
│   ├── src/
│   │   ├── api/               # 數據接入層
│   │   ├── indicators/        # 技術指標引擎
│   │   ├── backtest/          # VectorBT回測引擎
│   │   ├── strategies/        # 交易策略庫
│   │   └── utils/             # 工具函數
│   ├── config/                # 配置管理
│   ├── tests/                 # 測試套件
│   └── docs/                  # 文檔
├── archive/                    # 歸檔舊代碼
├── experiments/                # 實驗性功能
└── ops/                        # 運維腳本
```

### 2. 數據源標準化
建立統一的數據管理框架：

- **數據源認證**: 只使用經過驗證的真實API端點
- **質量控制**: 實施數據完整性檢查和異常處理
- **版本管理**: API接口版本控制和向後兼容
- **緩存策略**: 智能緩存機制，平衡性能與數據新鮮度

### 3. 性能優化策略
- **VectorBT優化**: 充分利用向量化計算能力
- **GPU加速**: 簡化GPU配置，實現自動檢測和回退
- **並行計算**: 優化多核CPU利用率
- **內存管理**: 改進數據結構和內存使用效率

## 預期收益

### 開發效率提升
- **文件數量減少**: 從400+文件精簡到~50核心文件
- **代碼重複率降低**: 目標減少80%的重複代碼
- **新功能開發**: 開發週期預期減少60%

### 系統可靠性增強
- **數據一致性**: 100%使用經過驗證的真實數據源
- **回測準確性**: 消除Mock數據導致的結果偏差
- **環境一致性**: 開發、測試、生產環境統一

### 性能顯著改善
- **策略優化速度**: 目標提升5-10倍
- **資源利用率**: CPU使用率從目前水平提升到80%+
- **GPU加速**: 在支持的硬體上實現10-50倍性能提升
