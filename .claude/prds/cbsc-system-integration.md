# PRD: CBSC系統統一整合

## Executive Summary

本PRD旨在解決CBSC量化交易系統當前面臨的三大核心問題：多套前端系統並存、後端服務架構分散、數據架構設計不規範。通過系統性的統一整合，建立現代化的企業級量化交易平台，實現技術債務清零、開發效率提升、系統穩定性增強。

## Problem Statement

### 核心問題識別

基於深入的系統分析，我們識別出三個根本性問題：

#### 🚨 問題1: 多套前端系統並存，技術棧重複
- **unified-dashboard**: React 18 + TypeScript + Vite + Ant Design 5.12.8 + Tailwind CSS
- **frontend**: React 18 + react-scripts + Chart.js 4.2.0 + Ant Design 5.5.0
- **strategy-dashboard**: HTML5 + Vanilla JavaScript + Chart.js
- **localhost_interface**: 另一套React前端系統

**影響**: 開發效率降低40%，維護成本高昂，用戶體驗不一致

#### 🚨 問題2: 後端服務架構分散，缺乏統一API網關
- **多個 FastAPI 服務**: simple_dashboard_server.py (Port 8080) + 其他微服務
- **多個數據源**: JSON文件 + PostgreSQL + Redis + CSV歷史存檔
- **缺乏統一認證**: 各服務獨立的用戶驗證

**影響**: 服務管理複雜，數據一致性問題，安全風險增加

#### 🚨 問題3: 數據架構設計不規範，數據一致性問題
- **混合存儲策略**: JSON文件系統 + PostgreSQL + Redis + CSV文件
- **數據流混亂**: CBSC牛熊證數據 + 6個香港政府API + 策略信號數據
- **缺乏數據治理**: 沒有統一的數據模型和版本控制

**影響**: 數據可靠性風險，性能瓶頸，擴展困難

## User Stories

### 主要用戶群體

#### 1. 系統開發者（主要用戶）
**用戶故事**: 作為系統開發者，我希望在一個統一的技術棧上工作，而不是在4套不同的前端系統之間切換，以便我能够高效地開發新功能。

**當前痛點**:
- 需要維護多套React配置和依賴
- 相同功能在不同系統中重複實現
- 代碼風格和架構模式不一致
- 新功能需要多地部署和測試

**驗收標準**:
- 單一前端代碼庫，統一技術棧
- 一次開發，全系統生效
- 統一的UI/UX設計規範

#### 2. 系統管理員
**用戶故事**: 作為系統管理員，我希望有一個統一的部署和管理界面，而不是監控多個分散的服務，以便我能確保系統穩定運行。

**當前痛點**:
- 需要監控多個端口和服務
- 認證和權限配置分散
- 日誌和監控數據碎片化
- 備份和恢復流程複雜

**驗收標準**:
- 統一的API網關入口
- 集中式的認證授權
- 統一的監控和日誌系統

#### 3. 量化分析師
**用戶故事**: 作為量化分析師，我希望數據準確一致，系統穩定可靠，以便我能專注於策略開發而不是處理系統問題。

**當前痛點**:
- 不同系統顯示的數據可能不一致
- 系統故障影響策略執行
- 數據導入導出流程複雜
- 歷史數據完整性問題

**驗收標準**:
- 數據一致性和準確性保證
- 系統可用性 > 99.9%
- 統一的數據訪問接口

## Requirements

### Functional Requirements

#### FR1: 前端系統統一 (Priority: P0)
- **FR1.1**: 選擇unified-dashboard作為主要前端框架
- **FR1.2**: 遷移其他系統的功能到統一技術棧
- **FR1.3**: 實施統一的UI設計系統 (Ant Design 5.12.8)
- **FR1.4**: 移除重複的前端代碼和依賴

#### FR2: 後端服務整合 (Priority: P0)
- **FR2.1**: 建立統一的API網關服務
- **FR2.2**: 整合FastAPI微服務，消除重複功能
- **FR2.3**: 實施統一的JWT認證授權機制
- **FR2.4**: 統一RESTful API設計規範

#### FR3: 數據架構重構 (Priority: P1)
- **FR3.1**: 統一數據存儲為PostgreSQL + Redis架構
- **FR3.2**: 遷移JSON/CSV數據到關係型數據庫
- **FR3.3**: 建立統一的數據模型和版本控制
- **FR3.4**: 實現數據一致性和完整性保障

#### FR4: 微服務架構升級 (Priority: P1)
- **FR4.1**: 按業務域拆分服務（策略管理、用戶管理、數據服務）
- **FR4.2**: 實現服務間通信機制（RESTful API + 消息隊列）
- **FR4.3**: 建立服務註冊和發現機制
- **FR4.4**: 實施負載均衡和容錯處理

### Non-Functional Requirements

#### NFR1: 性能要求
- **NFR1.1**: API響應時間 < 200ms (95%分位)
- **NFR1.2**: 頁面加載時間 < 2秒
- **NFR1.3**: 系統並發支持 > 100用戶
- **NFR1.4**: 數據庫查詢優化，支持大數據量處理

#### NFR2: 安全性要求
- **NFR2.1**: 統一的JWT + OAuth2認證機制
- **NFR2.2**: 基於角色的訪問控制(RBAC)
- **NFR2.3**: API訪問頻率限制和安全掃描
- **NFR2.4**: 敏感數據加密存儲和傳輸

#### NFR3: 可擴展性要求
- **NFR3.1**: 模塊化設計，支持插件式擴展
- **NFR3.2**: 微服務架構，支持獨立部署
- **NFR3.3**: 數據庫讀寫分離和緩存策略
- **NFR3.4**: 容器化部署支持(Docker + K8s)

#### NFR4: 可維護性要求
- **NFR4.1**: 代碼覆蓋率 > 80%
- **NFR4.2**: 技術債務評分 < B級
- **NFR4.3**: 平均bug修復時間 < 24小時
- **NFR4.4**: 完整的API文檔和開發文檔

## Constraints & Assumptions

### 約束條件
- **時間約束**: 整個整合週期不超過16週
- **資源約束**: 最多2-3名開發者並行工作
- **業務連續性**: CBSC核心功能必須保持可用
- **數據約束**: 歷史數據不能丟失，遷移過程零停機

### 技術假設
- 團隊具備React 18和FastAPI開發能力
- PostgreSQL數據庫性能滿足需求
- 現有基礎設施支持新架構部署
- CBSC數據接口保持穩定兼容

### 業務假設
- 用戶接受漸進式系統整合策略
- 整合過程中不影響日常交易操作
- 系統主要面向量化交易團隊使用

## Out of Scope

### 本次整合不包含的功能
- 完全重寫核心CBSC量化算法
- 支持多租戶SaaS模式
- 實現雲原生架構(Kubernetes)
- 國際化和多語言支持
- 移動端應用開發
- 第三方插件市場

### 延後的功能
- 高可用和災備方案
- 實時數據流處理(Kafka集成)
- 機器學習模型訓練平台
- 區塊鏈集成

- **測試團隊**: 需要制定整合測試策略

### 外部依賴

- **Redis**: 緩存和會話存儲
- **CBSC數據源**: 牛熊證數據接口穩定性

### 技術依賴

- **Week 9-10**: 數據庫整合，服務間通信機制實施，API網關建立

### Phase 4: 數據層統一 (3週)
- **Week 11-12**: 核心業務數據遷移，數據訪問層重構，數據一致性機制實施

### Phase 5: 部署和監控優化 (2週)

## Risk Assessment

### 高風險項
1. **數據遷移風險**: 歷史數據完整性問題
