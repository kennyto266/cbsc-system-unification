# PRD: CBSC 量化交易系統架構重構與優化

## Executive Summary

本 PRD 描述了 CBSC 量化交易策略管理系統的深度架構重構計劃。當前項目存在多個前端入口、重複的後端服務、嚴重的模塊耦合和分散的配置管理問題。本次重構將統一前端架構、簡化後端結構、優化依賴關係，並建立標準化的部署配置，以提升開發效率、降低維護成本、提高系統可擴展性。

**預計時程**: 1-2 個月（深度重構）

**核心價值**:
- 統一代碼庫，消除重複實現
- 解耦模塊，支持獨立開發和測試
- 標準化配置，簡化部署流程
- 更新文檔，保持架構與代碼同步

## Problem Statement

### 當前問題

#### 1. 前端架構分散
```
frontend/              # 主前端應用 (Vite + React)
unified-dashboard/     # 統一儀表板 (獨立項目)
strategy-dashboard/    # 策略儀表板 (子目錄)
```
- **問題**: 多個前端入口造成維護困難，代碼重複率高
- **影響**: 修復 bug 需要同步多個位置，功能不一致

#### 2. 後端架構重複
```
backend/               # FastAPI 後端服務
src/api/               # 另一組 API 實現
src/services/          # 業務服務
src/strategies/        # 策略實現
```
- **問題**: backend/ 和 src/ 存在功能重疊，模型定義重複
- **影響**: API 路由衝突，數據模型不一致

#### 3. 依賴關係複雜
```
frontend/src/ → backend/api/
frontend/src/ → src/api/
src/strategies/ → backend/api/
backend/models/ → src/models/
```
- **問題**: 模塊間循環依賴，邊界模糊
- **影響**: 修改一處可能影響多個模塊，難以單獨測試

#### 4. 配置分散
```
.env, .env.dev, .env.full, .env.prod (多處)
frontend/src/services/config.ts
backend/config/
src/core/config.py
docker-compose.yml, docker-compose.*.yml (多個版本)
```
- **問題**: 環境變量定義在不同位置，缺乏統一管理
- **影響**: 部署配置容易出錯，環境切換困難

#### 5. 文檔過時
- **問題**: 架構文檔與實際代碼不匹配
- **影響**: 新開發者上手困難，維護成本高

### 為什麼現在需要解決

1. **開發效率下降**: 每次修改需要同步多處代碼
2. **維護成本增加**: bug 修復和功能添加越來越困難
3. **新功能開發受阻**: 耦合嚴重，難以獨立開發新功能
4. **部署風險高**: 配置分散，生產部署容易出錯

## Requirements

### Functional Requirements

#### FR-001: 前端架構統一
**優先級**: P0 (必須)

**要求**:
1. 合併 `frontend/`、`unified-dashboard/`、`strategy-dashboard/` 為單一應用
2. 使用 React Router v6 管理所有路由
3. 建立共享的組件庫和狀態管理
4. 統一的 API 客戶端配置

**實現細節**:
```
新前端結構:
frontend/
├── src/
│   ├── app/              # 主應用入口
│   ├── features/         # 功能模塊
│   │   ├── strategies/   # 策略管理
│   │   ├── backtest/     # 回測系統
│   │   ├── realtime/     # 實時數據
│   │   └── dashboard/    # 統一儀表板
│   ├── shared/           # 共享資源
│   │   ├── components/   # 通用組件
│   │   ├── hooks/        # 共享 hooks
│   │   ├── services/     # API 服務
│   │   └── utils/        # 工具函數
│   └── config/           # 配置文件
```

#### FR-002: 後端架構簡化
**優先級**: P0 (必須)

**要求**:
1. 合併 `backend/` 和 `src/api/` 為統一 API 層
2. 整合數據模型定義到 `src/models/`
3. 統一業務服務到 `src/services/`
4. 清理重複的端點定義

**實現細節**:
```
新後端結構:
backend/
├── api/                  # API 端點 (統一)
│   ├── v1/               # API v1
│   └── v2/               # API v2
├── core/                 # 核心配置
├── models/               # 數據模型 (統一)
├── services/             # 業務服務
├── schemas/              # 請求/響應模式
├── middleware/           # 中間件
└── utils/                # 工具函數

src/
├── strategies/           # 策略實現 (保持)
├── backtest/             # 回測引擎 (保持)
├── trading/              # 交易執行 (保持)
└── websocket/            # WebSocket (保持)
```

#### FR-003: 依賴關係優化
**優先級**: P0 (必須)

**要求**:
1. 消除所有循環依賴
2. 建立清晰的模塊邊界
3. 實現依賴注入
4. 支持模塊獨立測試

**依賴規則**:
```
允許的依賴方向:
前端 → API Gateway → 服務層 → 數據層
禁止的依賴:
- 服務層 → 前端
- 數據層 → API Gateway
- 跨層級直接調用
```

#### FR-004: 配置管理統一
**優先級**: P0 (必須)

**要求**:
1. 單一的配置文件結構
2. 環境變量統一管理
3. 配置驗證和默認值
4. 敏感信息加密

**配置結構**:
```
config/
├── environments/
│   ├── development.yml
│   ├── staging.yml
│   └── production.yml
├── services/
│   ├── frontend.yml
│   ├── backend.yml
│   └── database.yml
└── shared/
    ├── api.yml
    └── features.yml
```

#### FR-005: API 版本管理
**優先級**: P1 (重要)

**要求**:
1. 實現 API 版本控制
2. 支持多版本並存
3. 漸進式棄用舊版本
4. API 文檔自動生成

#### FR-006: 測試覆蓋
**優先級**: P1 (重要)

**要求**:
1. 單元測試覆蓋率 > 80%
2. 集成測試覆蓋核心流程
3. E2E 測試覆蓋關鍵用戶旅程
4. 測試文檔與代碼同步

## Constraints & Assumptions

### 技術限制
1. **需要保持現有功能運行**: 重構期間不能中斷服務
2. **數據庫遷移複雜**: 需要謹慎處理數據模型變更
3. **向後兼容**: 需要支持現有 API 客戶端
4. **團隊熟悉度**: 團隊需要學習新架構

### 時間限制
1. **總時程**: 1-2 個月
2. **階段交付**: 每 1-2 週一個里程碑
3. **測試時間**: 佔總時程的 30%

### 資源限制
1. **開發團隊**: 2-3 名全棧開發者
2. **測試資源**: 1 名 QA 工程師
3. **DevOps 支持**: 部分時間
4. **產品經理**: 持續參與

### 假設條件
1. **現有功能需求不變**: 不添加新功能
2. **技術棧穩定**: 不更換主要框架
3. **數據遷移**: 可以接受短暫的停機時間
4. **用戶通知**: 可以安排維護窗口

## Out of Scope

本重構項目**不包含**以下內容：

### 功能性排除
1. **新功能開發**: 不添加新的策略或功能
2. **UI/UX 重新設計**: 不改變用戶界面外觀
3. **性能優化**: 除了架構相關的性能改進
4. **數據庫優化**: 不涉及數據庫結構調整

### 技術性排除
1. **雲服務商遷移**: 不改變部署平台
2. **監控系統重建**: 繼續使用現有監控
3. **CI/CD 重設計**: 只優化配置，不重建流程
4. **第三方服務整合**: 不添加新的外部服務

### 文檔性排除
1. **用戶手冊更新**: 不更新最終用戶文檔
2. **培訓材料**: 不製作培訓課程
3. **行銷內容**: 不涉及市場推廣內容
