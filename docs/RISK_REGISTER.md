## 1. 風險管理概述

### 1.1 風險評估方法

使用 **5x5 風險矩陣**評估每個風險:

```
                影響程度
    低      中      高      严重
    ┌────┬────┬────┬────┐
低  │  5 │ 10 │ 15 │ 20 │
    ├────┼────┼────┼────┤
中  │ 10 │ 15 │ 20 │ 25 │
    ├────┼────┼────┼────┤
高  │ 15 │ 20 │ 25 │ 30 │
    ├────┼────┼────┼────┤
极高 │ 20 │ 25 │ 30 │ 35 │
    └────┴────┴────┴────┘
      低  中  高  极高
      發生概率

風險分數 = 發生概率 × 影響程度

分數範圍:
- 5-14: 低風險 (綠色)
- 15-24: 中風險 (黃色)
- 25-35: 高風險 (紅色)
```

### 1.2 風險類別

| 類別 | 描述 | 示例 |
|------|------|------|
| **技術風險** | 技術實現相關 | 性能下降、兼容性問題 |
| **業務風險** | 業務影響相關 | 功能缺失、用戶流失 |
| **項目風險** | 項目管理相關 | 延期、資源不足 |
| **運營風險** | 運維相關 | 部署失敗、監控缺失 |
| **安全風險** | 安全相關 | 數據洩露、權限問題 |

### 2.1 技術風險

#### 風險 ID: T001 - 數據遷移失敗

| 屬性 | 值 |
|------|------|
| **描述** | 數據從舊結構遷移到新結構時失敗或數據損壞 |
| **類別** | 技術風險 |
| **發生概率** | 中 (3) |
| **影響程度** | 高 (5) |
| **風險分數** | **15** (中風險) |
| **階段** | 階段 3: 數據層整合 |
| **狀態** | 開放 |

**影響**:
- 用戶數據丟失
- 服務不可用
- 需要從備份恢復

**緩解措施**:
1. 在遷移前創建完整備份
2. 在測試環境驗證遷移腳本
3. 使用雙寫策略驗證數據一致性
4. 分批遷移，每批驗證
5. 準備詳細回滾程序

**應急計劃**:
- 立即停止遷移
- 從備份恢復數據庫
- 分析失敗原因
- 修復後重新遷移

**負責人**: 後端 Tech Lead
**審視日期**: 每週

## 4. 風險響應計劃

### 4.1 風險避免

**適用於**: 高影響、高概率風險

策略:
- 修改項目範圍避免風險
- 使用成熟技術而非實驗性技術
- 增加預算和時間

示例:
- T001: 分批遷移數據，每次遷移後充分驗證

### 4.2 風險緩解

**適用於**: 中高風險

策略:
- 降低發生概率
- 減少影響程度
- 轉移風險（保險）

示例:
- T002: 維護 API 版本兼容層
- O001: 實施藍綠部署

### 4.3 風險轉移

**適用於**: 可轉移風險

策略:
- 外包給專業團隊
- 購買保險
- 使用 SLA

示例:
- T005: 使用有 SLA 保證的雲服務

### 4.4 風險接受

**適用於**: 低風險或緩解成本高於影響

策略:
- 主動接受（記錄決策）
- 被動接受（不採取行動）
- 應急計劃就緒

示例:
- P003: 知識流失風險（已有文檔和交叉培訓）

## 5. 風險審視流程

### 5.1 定期審視

```mermaid
graph LR
    A[週一] --> B[風險狀態更新]
    B --> C[識別新風險]
    C --> D[評估風險分數]
    D --> E[更新緩解措施]
    E --> F[分配負責人]
    F --> G[記錄決策]

    style A fill:#e1f5ff
    style G fill:#e1ffe1
```

### 5.2 觸發審視

以下情況觸發緊急風險審視:
- 發生未預期的阻斷性問題
- 項目範圍或優先級變更
- 關鍵人員變動
- 技術堆疊變更
- 第三方服務變更
